# 为什么我给本地模型造了一个笼子

> Claude 是大脑,本地模型是手 —— 一个"反框架"编码 agent 的演进史。

这不是一篇"我做了个很牛的 autonomous agent"的吹嘘文。
恰恰相反,它讲的是我怎么**慢慢学会不信任 agent**,然后把它框进一个笼子里,反而让它干得又稳又省。

---

## 一、起点:我不想为改一行代码付费

事情很朴素。我每天大量的编码改动是机械的:加个字段、改个端口、给函数加段校验、按规格补个接口。这些事:

- 让云端强模型(Claude)亲自敲,要读一堆文件、输出整段代码、还得陪着反复试错——**全程烧 token、花钱**;
- 可它们又没难到需要那么强的脑子——难的是**想清楚要改什么**,不是**把它改下去**。

于是一个分工浮出水面:**想清楚(判断)交给强模型,改下去(执行)交给本地模型。** 强模型只出一行规格(几个 token),本地模型把读码、写码、试错这些烧 token 的体力活全包了——而它干活**不计入 Claude 的账**。省的就是 cc 的 token。

(注:这跟"数据不出机/隐私"无关——大脑是云端 Claude,代码上下文本来就过了它。本质就是 token 经济。)

这就是 `cc → Qwen-Agent` 的雏形:`cc`(Claude)当大脑,`Qwen-Agent`(本地模型)当手。

> **番外:省 token 其实有两翼。** Qwen-Agent 省的是"执行"端;后来我又接了 **codegraph**(代码索引 MCP),让 Claude **读懂代码**这一步也别烧 token——索引查询代替 read 一堆文件。左翼省"读"、右翼省"写",中间大脑只剩判断。(细节见 README)

---

## 二、第一版很天真:把任务丢给本地模型,放养

第一版我做得跟所有 agent 教程一样:给本地模型一堆工具,写个 ReAct 循环,丢个任务,让它自己跑。

然后就开始翻车。而且**不是翻在"不会写代码",是翻在别的地方**:

- 我让它"删掉 excel 导入相关代码",它**顺手把隔壁一个无关 controller 也改了**。
- 我让它改 A 文件,它 `grep` 来 `grep` 去绕了几十轮,**就是不下手**,一直烧到轮数上限。
- 它的 `edit` 老是失败——`old_string` 缩进记错一个空格就匹配不上,然后它**瞎猜着重试**,越试越离谱。
- 有时候它跑完说"已完成 ✅",我一看 `git diff`,**根本没改对,甚至没改**。

最气人的是最后这条:**它的自我报告不可信。**

---

## 三、撞墙:本地模型不是不会干,是"判断"会翻

我换了好几个模型想解决问题:gpt-oss、Qwen 的各种量化档(Q6、Q8、Q4)……

结论很反直觉:**在需要"判断"的任务上,这些本地中小模型几乎都会翻,而且翻的程度差不多。**

最典型的一次:一个"给符合条件的订单打折"的任务,正确答案是改 5 处,几个模型不约而同地**过度套用规则,改到了 9 处**。理由还说得头头是道。换更大的量化档?一样翻。

那一刻我想明白了一件事:

> **这不是"模型不够强"能解决的。判断类的活,本地中小模型就是不可靠。**

如果我继续赌"找个更强的模型",我会一直输。

---

## 四、转向:不赌模型,赌笼子

于是我把整个思路掉了个头:

**既然本地模型的"判断"不可信,那就别让它判断。**

- 判断(改哪几个、多义词指什么、概念怎么对应)→ **大脑(Claude)先做完,写进规格**,变成无歧义的事实。
- 本地模型 → 只做**无歧义的机械执行**:照着规格在指定位置改。
- 至于它执行时还可能犯的错(改错文件、改太多、改不中)→ **用纯确定性的代码护栏挡住或自动纠回**。

质量不再来自"模型多聪明",而来自三件确定的事:**好规格 + 硬护栏 + 机器验证。**

我把这句话刻在了项目顶上:**把不确定的智能,关进确定的笼子。**

---

## 五、笼子是一个坑一个坑垒出来的

护栏不是设计出来的,是**撞出来的**。每一道闸背后都有一次翻车:

| 翻车现场 | 长出来的护栏 |
|---|---|
| 删代码顺手改了无关文件 | `--allow` **写白名单**:范围外的 `edit`/`write` 工具层直接拒 |
| 在允许的文件里却改太多(逻辑越界) | `--max-diff-lines` **diff tripwire**:改动行数超阈值告警 |
| `grep`/`read` 绕圈几十轮不下手 | **anti-storm**:14 轮只探索不动手 → 判 `stuck` 早止损 |
| `edit` 的 `old_string` 老对不上、瞎猜 | **edit 未命中提示**:自动回最接近的现有内容(带行号),让它照着逐字重写 |
| 某轮空转,既不调工具也不给结论 | **nudge**:推一句让它继续,最多 N 次 |
| 不完整 tool call 漏进正文当成"完成" | **泄漏检测**:识别 `to=functions.xxx` 这类文本,不当完成 |
| 跑完说"完成了",其实没改对 | `--verify` **自动验证 + 自纠错**:验证失败把错误喂回去自己修,直到 PASS |

注意这些护栏的共同点:**它们都不依赖模型变聪明,全是确定性的代码。** 这正是它们可靠的原因。

其中我最喜欢 `edit` 的那个小设计——它体现了一条信条:**与其指望模型更聪明,不如让工具的反馈更友好(ACI)。** 模型 `old_string` 记错了?别只回"not found",直接把真实内容怼到它脸上,它下一次就对了。

---

## 六、模型可以随便换,框架一行不动

因为聪明的部分早就外包给大脑了,这层 agent 只是"框住的手",所以**换模型成本极低**——`--model` 改一下而已。

我自己的选型一路换过来:gpt-oss → Qwen3.6 的 Q6 → Q8 → 最后落在 **Q4**。
选 Q4 不是因为它"最强"(前面说了,judgment 任务它们都翻),而是因为:

- 它**最快、最省内存**(MoE 架构,3B 激活,4-bit);
- 反正质量靠 harness 兜底,那就在"都得 review"的前提下,**按速度和内存选**。

这就是"不赌模型"的好处:**模型是可替换的零件,笼子才是产品。**

> (附带一课:连模型的性能数字都得自己实测。我曾经差点信了某个"开了能提速 1.5x"的官方特性,实测发现对我这个量化模型反而**掉速一半**——还是那句,不信 self-report,自己测。)

---

## 七、几个反直觉的决定

- **没有 `finish` 工具**。"不再调工具、只回一段文本"本身就是完成信号。少给笨手一个要学的动作。
- **工具只有 6 个,而且输出故意截断**(grep 60 行、bash 3000 字符)。工具越多、上下文越大,本地模型越容易迷失。逼它精确定位,而不是大水漫灌。
- **non-stream**。流式在本地中小模型上更容易出半截 tool call、通道泄漏。非流式 + 标准格式,稳。
- **零三方依赖**。整个东西就一个 ~420 行的 Python 文件,标准库写成。没有 LangChain,没有 pip install。我要的是能看懂、能改、能信任的东西,不是一座黑箱。

---

## 八、这套适合谁

**适合**:

- 想用本地模型干**机械编码活**(改字段、加校验、按规格补接口),又不想被它乱改坑的人;
- 已经有一个强模型(Claude / GPT 等)当"大脑"出规格的工作流;
- 在意 **token 成本**的场景——尤其高频机械改动,省下的 cc token 很可观。

**不适合**:

- 指望它**自己想清楚要做什么**——它不替你判断,规格糊它就翻,这是设计取舍不是 bug;
- 用纯 completion、不支持 function calling 的模型。

---

## 尾声

我做这个项目最大的收获,不是"做出了一个 agent",而是想通了一件事:

> **面对一个不那么可靠的执行者,正确的做法不是逼它变可靠,而是设计一个让它的不可靠无所谓的系统。**

大脑负责聪明,护栏负责兜底,模型只是中间那双手——它可以换、可以不够强,只要笼子还在。

聪明留给大脑,稳健交给护栏。这就是 `cc → Qwen-Agent`。
